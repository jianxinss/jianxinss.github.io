React Native

A FRAMEWORK FOR BUILDING NATIVE APPS USING REACT

Created by 赵建鑫 / @James

what

React Native enables you to build world-class application experiences on native platforms using a consistent developer experience based on JavaScript and React. The focus of React Native is on developer efficiency across all the platforms you care about — learn once, write anywhere. Facebook uses React Native in multiple production apps and will continue investing in React Native.

  1. React Native可以让开发者在基于Javascript和React的开发经验基础之上开发一款与原生体验一致的应用。
  2. React Native核心放在解决的开发效率问题。learn once, write anywhere
  3. Facebook 将会利用React Native开发他们的应用，并继续投资于React Native。
What we really want is the user experience of the native mobile platforms, combined with the developer experience we have when building with React on the web.
With a bit of work, we can make it so the exact same React that's on GitHub can power truly native mobile applications. The only difference in the mobile environment is that instead of running React in the browser and rendering to divs and spans, we run it an embedded instance of JavaScriptCore inside our apps and render to higher-level platform-specific components.
It's worth noting that we're not chasing “write once, run anywhere.”
Different platforms have different looks, feels, and capabilities
and as such, we should still be developing discrete apps for each platform, but the same set of engineers should be able to build applications for whatever platform they choose, without needing to learn a fundamentally different set of technologies for each.
We call this approach
“learn once, write anywhere.”

当前移动应用的解决方案

  1. web流

①web流

hybrid

每个平台都有浏览器，都有webView 控件，我们可以使用HTML、CSS和javascript 将web 的内容和体验搬到本地。通过这么做，我们可以将逻辑 与UI渲染部分统一，以减少开发和维护成本，这种方式开发APP称之为Hybrid app。我们用的AppCan 、phoneGap和Cordova。这些框架一般都会放出相机、地理位置定位等调用硬件的接口

web 流产生的问题

1 性能慢

2 功能缺失

② <代码/> 转换流

将某个语言转成Object-C Javascript 或 C#，然后使用不同平台下地官方工具来开发。

  1. Object-C => Java
  2. java =>Object-C
  3. Java => C#
  4. hexe =>其他语言

③编译流

将某个语言编译成二进制文件，生成动态库，或者打包成apk/ipa/xap 文件

  • 优点 :重用显示复杂的代码 反编译困难 性能或许会更好
  • 缺点 :对bug 和 性能问题，定位和修改成本很高 编译后体积不小，尤其是支持Armv8 和x86的话

④虚拟机流

通过将某个语言的虚拟机移植到不同的平台

  • 性能损耗
  • 虚拟机占用空间

基于 PhoneGap 使用 JavaScript 和 HTML5 开发 iOS 应用已经有好几年了，那 React Native 有什么牛的？

VS 原生

  • 用户体验
  • 工程开发
  • App store 审核抗拒风险
  • Android版本时间推迟

VS phoneGap 和 Titanium

  • NO html NO WebView NO browser
  • React Native 要解决的是开发效率问题， 它并没指望完全取代 Native 开发，它的 rootView 继承自 UIView， 所以可以在部分 View 是使用，很方便混着，不需要重写整个 app， 而且混用的时候还需要显示地将 API 暴露给 JavaScript。
  • PhoneGap Titanium 那样企图完全使用 JavaScript 开发，将所有系统 API 都暴露给了 JavaScript， 让 JavaScript 语言默认就拥有 Native 语言的各种能力，然后再次基础上来开发

核心优势

  • 上手快，效率高，learn once，write anywhere.
  • React native 完全不用DOM， 最后呈现的是原生的效果。（No css 引起的reflow ,repaint）.
  • App实时更新，升级简单
  • 使用CSS FLexbox 对Native View 进行布局（我们有很熟悉的布局，有很多熟悉的样式， 当时不用担心泛滥的css style reflow）

More

  • 基于web技术，我们可以像在浏览器里那样，随时在仿真程序中查看应用的运行情况，刷新一下即可，无需编译
  • webComponent
  • 当你创建一些函数或对象时，用到一些类似的技术。其中一项技术就是单一功能原则，指的是，理想状态下一个组件应该只做一件事，假如它功能逐渐变大就需要被拆分成更小的子组件。

And

  • 作为前端开发，多掌握一些新技术，特别是大公司相对成熟的技术准会没错的。
  • 作为非前端开发，面对如此强大，有一统江湖趋势的Javascript，开始学习React总是会有益的。
  • 作为创业公司，前期需要做跨平台，迅速迭代的产品，React Native是值得去尝试的。
